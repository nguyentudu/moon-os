# Time To Contribution Measurement v1

## Definition

Time required for a new or returning contributor to make a useful contribution without repeating old mistakes.

## Why It Matters

Teams pay economic cost when contributors spend time rediscovering context, asking repeated questions, or making changes that violate prior decisions.

## Measurement Design

Compare:

- contributor using repository docs only
- contributor using Moon Decision Memory plus repository docs

## Contribution Task Examples

Use tasks that require real context, such as:

- propose where a new validation layer belongs
- decide whether Decision Memory should expand yet
- explain why ingestion is delayed
- identify whether a change violates Trust or Attention constraints
- recommend the next reliability step without reopening settled issues

## Captured Measures

- time to first useful proposal
- correctness of architecture placement
- evidence cited
- avoided rejected paths
- number of clarification requests
- reviewer confidence

## Useful Contribution Criteria

A contribution is useful when it:

- references relevant source evidence
- respects current constraints
- avoids known rejected paths
- explains tradeoffs
- can be reviewed by a maintainer without full rework
